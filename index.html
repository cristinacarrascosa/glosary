Internet (creation, concept, operation)

Creation: The origin of the Internet dates back to the 1960s. In the midst of the Cold War, the United States established a special military network in order to be able to access military information from anywhere in the country in a hypothetical Russian attack. The network was founded in 1969 and is called the ARPANET.

Concept: The Internet is the global system of interconnected computer networks that uses the Internet protocol suite (TCP/IP) to communicate between networks and devices. It is a network of networks that consists of private, public, academic, business, and government networks of local to global scope, linked by a broad array of electronic, wireless, and optical networking technologies

Operation: To do this, it uses WiFi wireless networks and Ethernet cables, which connect the devices with a router. The router is a device that intermediates between the devices and your Internet provider. And, in turn, the Internet provider provides you with Internet access thanks to its infrastructure of intermediate exchanges. For their part, these exchanges connect to each other and to millions of servers and other computers using kilometers and kilometers of land and submarine cables.

Types:

Transmission Control Protocol (TCP):
TCP is a popular communication protocol which is used for communicating over a network. It divides any message into series of packets that are sent from source to destination and there it gets reassembled at the destination.
Internet Protocol (IP): 
IP is designed explicitly as addressing protocol. It is mostly used with TCP. The IP addresses in packets help in routing them through different nodes in a network until it reaches the destination system. TCP/IP is the most popular protocol connecting the networks.
User Datagram Protocol (UDP):
UDP is a substitute communication protocol to Transmission Control Protocol implemented primarily for creating loss-tolerating and low-latency linking between different applications.
Post office Protocol (POP): 
POP3 is designed for receiving incoming E-mails.
Simple mail transport Protocol (SMTP): 
SMTP is designed to send and distribute outgoing E-Mail.
File Transfer Protocol (FTP): 
FTP allows users to transfer files from one machine to another. Types of files may include program files, multimedia files, text files, and documents, etc.
Hyper Text Transfer Protocol (HTTP):
HTTP is designed for transferring a hypertext among two or more systems. HTML tags are used for creating links. These links may be in any form like text or images. HTTP is designed on Client-server principles which allow a client system for establishing a connection with the server machine for making a request. The server acknowledges the request initiated by the client and responds accordingly.
Hyper Text Transfer Protocol Secure (HTTPS):
HTTPS is abbreviated as Hyper Text Transfer Protocol Secure is a standard protocol to secure the communication among two computers one using the browser and other fetching data from web server. HTTP is used for transferring data between the client browser (request) and the web server (response) in the hypertext format, same in case of HTTPS except that the transferring of data is done in an encrypted format. So it can be said that https thwart hackers from interpretation or modification of data throughout the transfer of packets.
Telnet:
Telnet is a set of rules designed for connecting one system with another. The connecting process here is termed as remote login. The system which requests for connection is the local computer, and the system which accepts the connection is the remote computer.
Gopher:
Gopher is a collection of rules implemented for searching, retrieving as well as displaying documents from isolated sites. Gopher also works on the client/server principle.

HTTP protocol (concept, utility, characteristics, structure of an HTTP request and response packet, HTTP methods and their utility)

Concept: is the communication protocol that allows information transfers through files on the World Wide Web

Utility: The http, as has been said, is a language that mediates between client requests and server responses on the Internet, to allow fluid communication and in the same "language". This protocol establishes the guidelines to follow, the request methods (called “verbs”) and has some flexibility to incorporate new requests and functionalities, especially as its versions progress.

HTTP requests and responses share a similar structure, composed of:

  1. A start line describing the request to be implemented, or its status, either success or failure. This start line is always a single line.
  2. An optional group of HTTP headers, indicating the request or describing the body that is included in the message.
  3. An empty line indicating all meta-information has been sent.
  4. An optional message body field that carries the data associated with the request (such as the content of an HTML form), or the files or documents associated with a response (such as an HTML page, or an audio, video... file) . The presence of the body and its size is indicated in the start line and HTTP headers.

HTTP methods and their utility

  • The GET method requests a representation of the specified resource. Requests using GET should only retrieve data.
  • The HEAD method asks for a response identical to a GET request, but without the response body.
  • The POST method submits an entity to the specified resource, often causing a change in state or side effects on the server.
  • The PUT method replaces all current representations of the target resource with the request payload.
  • The DELETE method deletes the specified resource.

  • The TRACE method performs a message loop-back test along the path to the target resource.
  • The PATCH method applies partial modifications to a resource.

TCP (concept, utility, well-known ports, structure of TCP segments, sockets)

Internet protocol family:



The transmission control protocol (TCP) is, like the UDP protocol such as SCTP, an Internet protocol that is located in the transport layer of the OSI model. The purpose of the TCP protocol is to create connections within a data network made up of computer networks to exchange data. In addition, in terms of its operation, it guarantees that the data will be delivered to its destination without errors and in the same order in which it was transmitted2. It also provides a mechanism to distinguish different applications within the same machine, through the port concept.

TCP utility:
In the TCP/IP protocol stack, TCP is the middle layer between the network protocol (IP) and the application. Applications often require reliable communication over the network. For this, the TCP protocol is implemented, which ensures that the data issued by the client are received by the server without errors and in the same order that they were issued, despite working with the services of the IP layer, which is not reliable. It is a connection-oriented protocol, since the client and the server must announce each other and accept the connection before starting to transmit the data to that user who must receive it.TCP supports many of the most popular Internet applications (browsers, file sharing, FTP clients, etc.) and HTTP, SMTP, SSH, and FTP application protocols.

TCP segment format: At the transportation level, the packets of bits that constitute the data units of TCP protocol are called "segments".

IP (concept, utility, IP addresses, routing, structure of IP datagrams)

An Internet Protocol address (IP address) is a numerical label such as 192.0.2.1 that is connected to a computer network that uses the Internet Protocol for communication.[1][2] An IP address serves two main functions: network interface identification and location addressing.

An IP address serves two principal functions: it identifies the host, or more specifically its network interface, and it provides the location of the host in the network, and thus the capability of establishing a path to that host. Its role has been characterized as follows: "A name indicates what we seek. An address indicates where it is. A route indicates how to get there."[2] The header of each IP packet contains the IP address of the sending host and that of the destination host.

IPv4 addresses An IPv4 address has a size of 32 bits, which limits the address space to 4294967296 (232) addresses. Of this number, some addresses are reserved for special purposes such as private networks (~18 million addresses) and multicast addressing (~270 million addresses). IPv4 addresses are usually represented in dot-decimal notation, consisting of four decimal numbers, each ranging from 0 to 255, separated by dots, e.g., 192.0.2.1. Each part represents a group of 8 bits (an octet) of the address.[8] In some cases of technical writing,[specify] IPv4 addresses may be presented in various hexadecimal, octal, or binary representations.

IPv6 addresses In IPv6, the address size was increased from 32 bits in IPv4 to 128 bits, thus providing up to 2128 (approximately 3.403×1038) addresses. This is deemed sufficient for the foreseeable future. The large number of IPv6 addresses allows large blocks to be assigned for specific purposes and, where appropriate, to be aggregated for efficient routing. With a large address space, there is no need to have complex address conservation methods as used in CIDR. All modern desktop and enterprise server operating systems include native support for IPv6, but it is not yet widely deployed in other devices, such as residential networking routers, voice over IP (VoIP) and multimedia equipment, and some networking hardware.

Routing IP addresses are classified into several classes of operational characteristics: unicast, multicast, anycast and broadcast addressing.

IP Datagram General Format The IPv4 datagram is conceptually divided into two pieces: the header and the payload. The header contains addressing and control fields, while the payload carries the actual data to be sent over the internetwork. Unlike some message formats, IP datagrams do not have a footer following the payload.

URL (concept, structure, parameters)

Uniform Resource Locator, URL parameters or query strings are the part of a URL that typically comes after a question mark (?) and are used to pass data along with the URL. They can be active parameters that modify page content or passive parameters that are mostly used for tracking and do not change the content.

They are made up of key-value pairs, where the key tells you what data is being passed and the value is the data you’re passing, such as an identifier. They look like ?key=value but may be separated by ampersands (&) like ?key=value&key2=value2 if there is more than one pair.

How parameters are used

  • Filter. qRemoves some of the content, leaving more specific content on the page that a user wants to see. An example of this is faceted navigation in e-commerce.
  • Sort. Reorders the content in some way, such as by price or rating.
  • Paginate. Divides content into a series of related pages.
  • Translate. Changes the language of the content.
  • Search. Queries a website for information that a user is looking for.
  • Affiliate IDs. Passes an identifier used to track where sales and signups come from.
  • Session IDs. Identifies a particular user. It’s not common on modern websites to use session IDs to track users.
  • Video timestamps. Jumps to the designated timestamp in a video.

Client-server applications vs P2P

Client-server:

In a client server network, there are clients and servers. A client can be a device or a program. It helps the end users to access the web. Some examples of clients are desktop, laptops, smartphones, web browsers, etc. A server is a device or a program that responds to the clients with the services. It provides files, databases, web pages, shared resources according to its type.



P2P:

In a peer to peer network, there is no specific client or a server. A device can send and receive data directly with each other. Each node can either be a client or a server. It can request or provide services accordingly. A node is also called a peer.

What is a cookie and what is it for? How can we establish a session using cookies? where does the cookie travel within the HTTP packet?

HTTP cookies (also called web cookies, Internet cookies, browser cookies, or simply cookies) are small blocks of data created by a web server while a user is browsing a website and placed on the user's computer or other device by the user's web browser.

What is it for?:

  • To keep track of which user is assigned to which shopping cart, the server sends a cookie to the client that contains a unique session identifier (typically, a long string of random letters and numbers). Because cookies are sent to the server with every request the client makes, that session identifier will be sent back to the server every time the user visits a new page on the website, which lets the server know which shopping cart to display to the user.
  • Another popular use of cookies is for logging into websites. When the user visits a website's login page, the web server typically sends the client a cookie containing a unique session identifier. When the user successfully logs in, the server remembers that that particular session identifier has been authenticated and grants the user access to its services.


How can we establish a session using cookies?:

The session cookie is a server-specific cookie that cannot be passed to any machine other than the one that generated the cookie. The server creates a “session ID” which is a randomly generated number that temporarily stores the session cookie. This cookie stores information such as the user’s input and tracks the movements of the user within the website. There is no other information stored in the session cookie.



where does the cookie travel within the HTTP packet?

Cookies are set using the Set-Cookie header field, sent in an HTTP response from the web server. This header field instructs the web browser to store the cookie and send it back in future requests to the server (the browser will ignore this header field if it does not support cookies or has disabled cookies).

What are MIME types and what are they for?

A media type (also known as a Multipurpose Internet Mail Extensions or MIME type) indicates the nature and format of a document, file, or assortment of bytes. MIME types are defined and standardized in IETF's RFC 6838.

Types: discrete and multipart. Discrete types are types which represent a single file or medium, such as a single text or music file, or a single video. A multipart type is one which represents a document that's comprised of multiple component parts, each of which may have its own individual MIME type; or, a multipart type may encapsulate multiple files being sent together in one transaction. For example, multipart MIME types are used when attaching multiple files to an email.

Software development methodologies:

Cascade model

The cascade model is a sequential development approach, in which development is seen as flowing steadily downwards (like a cascade) through several phases, typically:

  • Requirements
  • Software Design
  • Implementation
  • Testing
  • Integration, if there are multiple subsystems
  • Deployment (or Installation)
  • Maintenance

Integration testing concept (concept, tools)

Integration testing (sometimes called integration and testing, abbreviated I&T) is the phase in software testing in which individual software modules are combined and tested as a group. Integration testing is conducted to evaluate the compliance of a system or component with specified functional requirements.[1] It occurs after unit testing and before system testing. Integration testing takes as its input modules that have been unit tested, groups them in larger aggregates, applies tests defined in an integration test plan to those aggregates, and delivers as its output the integrated system ready for system testing.

Program documentation (concept, types, tools)

Concept:
Software documentation is a way for engineers and programmers to describe their product and the process they used in creating it in formal writing. Early computer users were sometimes simply given the engineers' or programmers' notes. As software development became more complicated and formalized, technical writers and editors took over the documentation process.

Types:
The two main types of software documentation are internal and external.

  • Internal documentation is used by the developers and other people who work on the software. It is usually written in a format that is easy for the developers to read and edit. Internal documentation is not intended for the end user.
  • External documentation is intended for the end user. It is usually written in a format that is easy for the user to read and understand.



Software documentation tools:
Various tools help vendors and developers automate the documentation process. Some important features of leading software documentation tools include the following:

Normalization is a set of rules that allow us to ensure that a relational schema meets certain properties, avoiding:

4. Insertion anomalies: impossibility to add data in the database due to the absence of other data.

In relational database theory, the forms normal (FN) provide the criteria for determining the degree of vulnerability of a table to inconsistencies and logical abnormalities. How much more high is the normal form applicable to a table, less vulnerable it will be to inconsistencies and anomalies. Edgar F. Codd originally defined the first three normal forms (1NF, 2NF, and 3NF) in 1970. The fourth and fifth normal forms (4NF and 5NF) deal specifically with representation of many-to-many and one-to-many relationships many among the attributes and were introduced by Fagin in 1977 and 1979 respectively.Each normal form includes the previous ones.

A Relationship is in 1NF if and only if each attribute is atomic.

A Relationship is in 2NF if and only if it is in 1NF and all the attributes that are not part of the Primary Key have full functional dependency on she.

Third Normal Form: 3NF
A Relationship is in 3NF if and only if it is in 2NF and there are no transitive dependencies. All functional dependencies must be on the key major.

Normalization is a set of rules that allow us to ensure that a relational schema meets certain properties, avoiding:
1. Data redundancy: repetition of data in a system.
2. Update anomalies: inconsistencies in the data as a result of redundant data and updates partials.
3. Deletion anomalies: unintentional loss of data data because other data has been erased.
4. Insertion anomalies: impossibility of adding data in the database due to the absence of other data.

In relational database theory, the shapes normal (FN) provide the criteria for determining the degree of vulnerability of a table to inconsistencies and logical abnormalities. How much more high is the normal form applicable to a table, minus vulnerable it will be to inconsistencies and anomalies. Edgar F Codd originally defined the first three normal forms (1NF, 2NF, and 3NF) in 1970. These forms Standards have been summarized as requiring that all attributes are atomic, depend on the full key, and direct (not transitive) form. Boyce–Codd normal form (FNBC) was introduced in 1974 by the two authors that appear in its name. the fourth Y fifth normal forms (4NF and 5NF) deal specifically of representation of many-to-many and one-to-many relationships many among the attributes and were introduced by Fagin in 1977 and 1979 respectively.Each normal form includes the previous ones.
